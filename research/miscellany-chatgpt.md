# Stopping Criteria  
Several approaches can govern when to stop querying. A simple rule is a *fixed budget*: ask, say, *k × N* comparisons. In practice, theory suggests **O(N log N)** comparisons suffice to recover the ranking under mild noise, so choosing *k* on the order of a few (e.g. 2–5) is reasonable for N=20–80. For example, Heckel *et al.* show near‐optimal performance up to log factors. However, a fixed budget is inflexible; it’s safest to allow the user to stop any time. By the Bayesian *stopping-rule principle*, inference is unaffected by how we choose to stop. In other words, we can safely halt early (or let the user stop) and simply report the current posterior ranking without bias.

More adaptive criteria can exploit the posterior. One can stop when a global uncertainty measure is low: for example, when all items’ credible‐interval widths fall below some threshold or when **posterior probability of correct ordering** exceeds, say, 95%. In ranking problems this is analogous to “posterior probability of correct selection” (pPCS) or “posterior probability of good selection” (pPGS) stopping rules. Concretely, one might stop when every adjacent pair in the sorted list is well‐separated (e.g. nonoverlapping 95% intervals) or when the expected “surprise” or uncertainty remaining is negligible. 

A **stability criterion** is also effective: stop when the inferred ranking stops changing. For instance, if the top‐K order or full ranking has remained identical (or within a small distance) for the last *m* answers, one is at a *stable state*. Cai *et al.* formalize this: the *optimal stopping point* is when the process enters a stable state (no appreciable rank changes thereafter). In practice one can check after each answer whether the top of the list shifts; if not for, say, 10 successive comparisons, halt.

Finally, one can stop when **marginal information gain** from the next question is below a threshold. Since we select pairs by expected information gain, we can track the highest gain available at each step. If the maximum expected KL‐gain (in bits) falls below ε, further queries add little. This ties into the posterior-based view: low expected gain means the posterior is already sharp. In summary, we recommend **combining criteria**: set a generous cap (e.g. ~3–5N), but allow early stopping when either (a) credible intervals are tight or (b) ranking is stable or (c) extra info gain is negligible. In all cases the user can stop at any time and the Bayesian posterior still yields a sensible ranking.

# Progress Indicator  
To show “how far along” the user is, we need a proxy for remaining uncertainty. A natural measure is the **total posterior entropy** of the scores. As comparisons accumulate, the joint entropy (or sum of individual variances) should drop; one can normalize this (e.g. relative to the start) and drive a progress bar. Another proxy is the count of **“uncertain pairs”** – for example, count how many adjacent items in the current ranking have win‐probabilities near 50% (or overlapping CIs). As this number falls to zero, the ranking is essentially fixed. Practically, one could compute for each consecutive pair (i,j) the probability P(i>j) under the posterior (e.g. using mean scores) and flag it “uncertain” if 40–60%. Finally, tracking **Kendall‐tau stability** may help: measure the rank‐correlation of the current ordering versus the previous step or versus the MAP estimate. Convergence (progress) is indicated by small changes.

No single proxy perfectly equals “remaining questions,” so it’s best to display a *fused progress metric*. For example, a slider labeled “Confidence” might combine fraction of uncertainty reduced (posterior entropy) and stability of the current ranking. Empirically, we found that posterior entropy correlates well with actual error, so reporting something like “96% confident” (normalized entropy) can resonate with users. Because users are non-technical, avoid raw probability jargon; instead use a colored bar or percentage that rises as more certainty is gained. For instance, when nearly all credible intervals become small, show a green “80–100%” confident status. In sum: **use a progress bar or percentage tied to posterior uncertainty (or an equivalent confidence index)**, rather than just “questions answered / total.” 

# Tiered (Grouped) Output  
Once finished, we group items into tiers so that nearly tied items aren’t split arbitrarily. A principled approach is to merge any items whose score estimates’ credible intervals overlap.  If item A’s 95% interval overlaps item B’s interval, treat A and B as effectively equal in rank.  Formally, define a dominance relation: A **dominates** B if A’s entire interval is above B’s (no overlap); otherwise the order is uncertain. In practice, sort items by posterior mean and then scan: whenever the CI of item *k* overlaps that of item *k+1*, keep them in the same tier (taking the transitive closure of overlaps).  This automatically yields fewer tiers when data are noisy. For example, Wicklin suggests plotting each item with its CI (“error bars on the ranked items”) to highlight where ranks are indistinguishable. In code, one can simply cluster consecutive items until gaps exceed uncertainty.

Alternatively, classical classification methods can be applied: *natural breaks* (Fisher–Jenks) or 1D *k-means* on the score estimates. These find large gaps in the sorted scores. Jenks finds “breaks” minimizing within-tier variance, which may coincide with big gaps. K-means (with *k* chosen by some criterion) also groups similar scores. However, these methods ignore uncertainty unless modified. For small N, the overlap rule above is easiest and uncertainty-aware. If a user prefers a fixed number of tiers, one can still run k-means or Jenks on the means to cut into that many groups and then **expand tiers** to merge any near-ties (i.e. merge an item with the tier above if its CI overlaps). 

In summary, we recommend *CI‐overlap grouping* by default: items with overlapping credible intervals go in the same tier. This respects uncertainty and avoids splitting what are statistically ties. If automated tier count is needed, start with these connected components of overlap; if too coarse, the user can manually split a tier or vice versa. Always label tiers (e.g. “Tier 1 (Top), Tier 2 …”) and, ideally, display each item’s score±CI so users see which tiers are certain or not.

# Displaying Uncertainty / Confidence  
Communicating uncertainty is crucial. The simplest visual is to draw each item’s score with an **error bar or shaded interval**. For example, plot the posterior mean on the x-axis and draw a horizontal bar for its 95% credible interval. (Wicklin notes that “the simplest approach is to include confidence intervals (error bars) on a plot of the ranked items”.) These bars immediately show which items are well-separated and which overlap. Style them clearly but not obtrusively: use thin, neutral‐gray lines or bands (so as not to confuse with the score bars themselves). Always **label the CI** (e.g. “95% CI”) or explain it in a caption so non-expert users understand these lines are uncertainty.

Beyond error bars, use intuitive icons or colors for “confidence.” For example, you might display a green check mark or bar for items whose rank is almost certain (say >90% probability of beating the next item), yellow/orange if it’s moderate, and maybe a red warning for very close calls. (Design guidelines suggest using a consistent color scheme: e.g. green = high confidence, yellow = medium, red = low.) In text, avoid statistical jargon – say “A > B with 95% certainty” or simply “Almost tied” instead of “50% win probability.” A tooltip on an item can give the numeric probability or interval if the user wants details. You could also highlight “uncertain pairs”: for instance, if P(i>j)≈0.5 for adjacent items, draw a dashed connector or a “?” icon between them. The key is to show *where* in the list uncertainty remains. Overall, combining error bars with color-coded confidence levels (green/orange/red) and clear labels will make the output comprehensible to lay users. 

# Handling Intransitive Answers  
Human answers may cycle (A > B, B > C, C > A) due to mistakes or real intransitivity. The Bradley–Terry model by construction finds the best one-dimensional fit to such data, effectively “averaging out” small cycles. In practice, mild intransitivity is handled by the model’s likelihood: it won’t represent the cycle explicitly but will produce the ranking that minimizes total surprise (total log-loss) across all answers. 

For larger inconsistencies, the interface can either **flag or re-ask**. One approach is to compute a simple inconsistency metric (e.g. HodgeRank cycle residuals) and, if it’s unusually high, prompt the user: “Your answers conflict (we found a cycle). Would you like to double-check one of these comparisons?” Often you’d re-ask the weakest-link comparison (the one with P ≈ 0.5). Alternatively, the system can down-weight any “outlier” answers in the Bayesian update (many robust BT extensions do this) or simply leave the posterior to dilute that answer. As a UI convention, you could highlight any triad that cycles (e.g. coloring a triangle icon), but most users just want a final ranking. In summary, minor cycles can be ignored (the model handles them). If serious contradictions appear, consider re-asking those specific pairs or explicitly acknowledging: “Ranking may be unreliable due to conflicting comparisons.”
